# Agent Teams

이 디렉토리는 agent team 운영 기준을 관리한다.

## 팀 구성
- `design-team`: 요구사항을 설계 결정으로 고정
- `implement-with-tdd-team`: 구현 + 테스트 동시 작성
- `review-team`: 아키텍처/품질 리뷰
- `test-team`: 시나리오/E2E/빌드 최종 검증

## 팀별 진입 문서
- `design-team/00_INDEX.md`
- `implement-with-tdd-team/00_INDEX.md`
- `review-team/00_INDEX.md`
- `test-team/00_INDEX.md`

## 공통 운영 원칙
- 모르면 반드시 질문한다. 추측으로 진행하지 않는다.
- 주요 결정에는 대안/트레이드오프/선택 이유를 남긴다.
- 단계별 산출물 없이 다음 단계로 넘어가지 않는다.
- 설계 없는 구현 금지
- 문서 위반 상태 merge 금지
- 정책값 변경 시 `capabilities/detail`과 `adr` 동시 업데이트
- 팀 산출물은 00_INDEX의 입출력 정의를 따라야 한다.

## 공통 DoD
- 입력/출력이 명시되어 있고 추적 가능한 위치가 있다.
- 미해결 질문은 미결사항 목록으로 남긴다.
- 다음 Gate 조건을 만족한 뒤에만 다음 팀으로 넘긴다.
- 결정은 ADR 또는 팀별 산출물로 근거가 추적 가능해야 한다.

## 팀별 역할 요약

### design-team
- 책임: 요구사항을 아키텍처 결정으로 변환
- 입력: 요구사항, 제약사항, 기존 문서
- 출력: 설계 결정서, 리스크, 구현 체크리스트
- DoD: 경계/트레이드오프/작업분해 명확화

### implement-with-tdd-team
- 책임: 설계 결정을 코드와 테스트로 구현
- 입력: 설계 결정서, 관련 문서
- 출력: 코드 변경, 테스트, 문서 동기화
- DoD: 신규 테스트 통과, 회귀 없음, 규칙 위반 없음

### review-team
- 책임: 품질/아키텍처 준수 점검
- 입력: PR, 테스트 결과, 문서 기준
- 출력: 리스크/위반 항목/수정 요청
- DoD: 치명 리스크 해소, 머지 가능 여부 명확화

### test-team
- 책임: 머지 직전 최종 검증
- 입력: 리뷰 통과 후보 브랜치, 테스트 시나리오
- 출력: pass/fail 리포트, 실패 재현 절차
- DoD: 필수 검증 통과, 배포 차단 이슈 없음

## Phase Gate (필수 통과조건)

### Gate 0: Design -> Implement
- 설계 범위, 레이어 경계, 리스크, 미결사항이 문서화되어 있어야 한다.
- 미결사항은 "질문 목록"으로 명시되어야 한다.

### Gate 1: Implement -> Review
- 코드 + 테스트가 함께 제출되어야 한다.
- 실패/취소/에러 경로 테스트가 포함되어야 한다.

### Gate 2: Review -> Test
- BLOCKER 이슈가 0건이어야 한다.
- WARNING 이슈는 추적 가능한 TODO/티켓으로 남겨야 한다.

### Gate 3: Test -> Merge
- 필수 테스트/빌드/E2E 검증이 통과해야 한다.
- 실패 항목은 재현 절차와 원인 분석이 있어야 한다.

## 기본 플로우
1. design-team이 설계 결정을 고정한다.
2. implement-with-tdd-team이 코드와 테스트를 작성한다.
3. review-team이 문서 기준으로 위반/리스크를 점검한다.
4. test-team이 최종 검증 후 pass/fail을 결정한다.

## 실행 전 체크 (새 프로젝트 공통)
- 과제 작업을 스크립트화하려면 tuist 설치가 필요하다.
- tuist가 없으면 기본 메시지로 설치 가이드를 제공하고 진행을 중단한다.
- 자동 설치를 허용하려면 `TUIST_AUTO_INSTALL=true` 환경변수를 설정한다.
