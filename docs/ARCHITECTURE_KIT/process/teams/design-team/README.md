# Design Team

## 역할
요구사항을 구현 가능한 설계 결정으로 변환한다.

## 입력
- 기능 요구사항
- 제약사항(일정, 리스크, 호환성)
- 기존 문서(`core`, `ui-patterns`, `capabilities`)

## 출력
- 설계 결정 요약(무엇/왜/범위)
- 변경 대상 모듈/레이어
- 리스크와 대응 전략
- 구현팀 전달 체크리스트
- 미결사항(질문 목록)

## 필수 원칙
- 모르면 반드시 질문한다. 추측으로 진행하지 않는다.
- 설계 결정마다 이유를 남긴다.
- 대안이 있을 경우 트레이드오프를 함께 기록한다.

## 분석 체크포인트
- 요구사항 누락/과잉/모순 점검
- 영향 범위(모듈/호환성/데이터) 점검
- 엣지 케이스 점검:
  1. 동시성/경쟁 상태
  2. 부분 실패
  3. 중복 요청/멱등성
  4. 상태 전이 충돌
  5. 외부 시스템 장애
  6. 입력 경계값/유효성
  7. 시스템 간 정합성

## 산출물 템플릿
```md
## 설계 결정
- 목표:
- 범위:
- 비범위:

## 핵심 결정
| 결정 | 대안 | 선택 이유 | 트레이드오프 |
|------|------|-----------|--------------|

## 영향 범위
| 모듈 | 영향 내용 | 위험도 |
|------|-----------|--------|

## 미결사항 (질문 필요)
- [ ] 질문 1
- [ ] 질문 2
```

## 완료조건 (DoD)
- 경계(레이어/의존성)가 명확하다
- 트레이드오프가 기록되어 있다
- 구현 가능한 단위로 작업이 분해되어 있다
- 미결사항이 질문 목록으로 명시되어 있다
